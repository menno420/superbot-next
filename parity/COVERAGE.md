## Methodology

- **Denominators are live**: command lists from `bot.walk_commands()` / `bot.tree.walk_commands()` on the booted bot; panel components from `core.runtime.panel_manifest.build_panel_manifest()`; events from `KNOWN_EVENTS`; tables from `pg_tables`.
- **Captured** means: at least one golden drives the surface (command invoked, custom_id rendered in a captured component tree, event observed on the bus tap, table appearing in a DB delta).
- **Coverage is breadth, not depth**: one captured invocation per command (admin persona, synthesized args) plus curated multi-step flows. A command with rich flag/subpath behavior counts as covered after one path — the per-subsystem coverage notes required at every `pending → ported` flip (design spec §10.1) are where depth gets audited.
- **Usage weighting is not applied**: the production usage counter (`command_total{cog,command,result}`) is an in-process Prometheus metric; no historical scrape exists in this environment. Weighted coverage needs the Phase-0.5 telemetry sidecar (design spec §2.10.4) — flagged as follow-up, not silently faked.
