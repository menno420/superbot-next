### R-1 — Land the scoped feature / port-surface backlog behind the audited seams
**Rationale.** The remaining user-visible parity gaps vs. live `superbot` are
already analyzed and PR-ready: blackjack + RPS-tournament remaining surface,
the reaction-adapter seam, and "the small four"
([`docs/ideas/*`](../ideas/)), each a replay-to-green slice behind the audited
domain seams (`sb/domain/<key>`). **Size: M** (several contained slices).
**Prod-readiness:** closes the last functional gaps so the new bot is a true
drop-in for the live one, not a near-parity fork.

### R-2 — Outbox + Discord-delivery retry/backoff with a dead-letter path
**Rationale.** `sb/kernel/outbox/{relay,store,enqueue}.py` is the delivery seam;
harden it with bounded exponential backoff + jitter on transient gateway/HTTP
failures and a dead-letter store for exhausted sends, surfaced via
`outbox/metrics.py`. **Size: M.** **Prod-readiness:** a momentary Discord API
blip or 429 must not silently drop a user-facing effect — retried-then-parked
beats lost-and-invisible.

### R-3 — DB pool reconnect + fail-closed degradation on Postgres loss
**Rationale.** `sb/kernel/db/pool.py` currently assumes a live connection; add
reconnect-with-backoff, a circuit breaker, and a fail-closed path so a Postgres
flap degrades to "politely refuse the write" rather than crashing the process or
half-applying an effect (the idempotency ledger in `db/idempotency.py` is the
anchor for safe replay). **Size: M.** **Prod-readiness:** managed Postgres
*will* drop connections (failover, maintenance); the bot must ride it out.

### R-4 — Metrics + readiness surface wired end-to-end
**Rationale.** `sb/kernel/observability/metrics.py` and `alerts.py` exist but the
HTTP surface is only `sb/adapters/http/health.py` (liveness). Add a Prometheus
scrape endpoint and a real `/readyz` that checks DB + gateway + outbox depth, so
the Railway service (NEXT-TASKS #4) can gate traffic on readiness. **Size: M.**
**Prod-readiness:** you cannot run, roll out, or alert on a prod service whose
internal state is invisible; readiness is what makes a safe cutover mechanical.

### R-5 — Property-based + fuzz coverage for the parsers and effect legs
**Rationale.** Golden parity (523/523) proves *known* inputs replay identically;
it does not find the malformed-command or adversarial-argument crasher. Add
Hypothesis property tests to the command/argument parsers and the effect-leg
compensation paths (the gaps already scoped in
[`effect-leg-compensation-gaps`](../ideas/effect-leg-compensation-gaps-2026-07-10.md)
and [`ensure-only-registration-gaps`](../ideas/ensure-only-registration-gaps-2026-07-10.md)).
**Size: M.** **Prod-readiness:** real Discord input is hostile and weird;
fuzzing finds the crash before a user does.

### R-6 — Migration + backup/restore + rollback drill, proven green in CI
**Rationale.** The pieces exist —
[`rollback-playbook.md`](../operations/rollback-playbook.md),
[`cutover-runbook.md`](../operations/cutover-runbook.md),
`restore-verify.yml`, `sb/kernel/db/migrations.py` — but the full
migrate → backup → restore → rollback loop has not been rehearsed end-to-end.
Turn on the data safety net (NEXT-TASKS #5) and add a CI leg that restores a
backup into a throwaway DB and asserts row-level integrity. **Size: L.**
**Prod-readiness:** an untested restore is a hope, not a recovery plan;
recoverability only counts once it has gone green unattended.

### R-7 — Secret-rotation runbook + startup credential fail-closed
**Rationale.** `sb/kernel/credentials/` and
[`credential-lifecycle.md`](../operations/credential-lifecycle.md) cover storage;
add a documented zero-downtime rotation procedure for `DISCORD_BOT_TOKEN_*` and
`DATABASE_URL`, plus a startup guard that fails closed (refuses to boot) on a
missing/malformed secret rather than half-starting. **Size: S.**
**Prod-readiness:** you must be able to rotate a leaked token in minutes, and a
bad secret must never produce a zombie half-booted process.

### R-8 — Structured-log audit: coverage, redaction, correlation IDs
**Rationale.** `sb/kernel/observability/redaction.py` and `findings.py` exist;
audit that every effect path emits a structured log line, that
`redaction.py` actually covers tokens/PII on all of them, and add a
per-interaction correlation ID threaded from adapter through effect so an
incident is traceable end-to-end. **Size: S.** **Prod-readiness:** incidents are
debugged from logs; unstructured or leaky logs make prod both unobservable and a
compliance liability.
