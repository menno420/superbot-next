# `manifest.snapshot.json` merge conflicts — the stable_hash fix + residual recipe

> **Status:** `binding` (runbook)

## The friction (historical)

`manifest.snapshot.json` is the committed K2 interchange artifact,
regenerated by `tools/manifest_compile.py --write`. Through 2026-07-13 it
carried a cached `stable_hash` line — a sha256 over the whole snapshot body.
Because that one line changed on EVERY recompile, any two concurrent PRs
that both recompiled the snapshot textually conflicted on it, even when
their body hunks were disjoint and git merged the 59k-line body cleanly
(hit PRs #333 and #352 on 2026-07-13; earlier: flip-playbook trap 10(e)).

## The fix (this runbook's reason to exist)

The hash membership (frozen spec 01 §5, fork 9) always EXCLUDED
`stable_hash` from the hashed body — the field was a pure cache of a value
derivable from the rest of the file. So the compiler simply **no longer
writes it**:

- `tools/manifest_compile.py` emits the snapshot WITHOUT a `stable_hash`
  field; the hash still exists in-memory (`CompileResult.stable_hash`) and
  in the tool's console output.
- P9 recompile-parity (and boot leg A, which wraps it) recompute the
  committed body's hash on the fly via `compute_stable_hash(committed)` and
  compare it against the source recompile. `compute_stable_hash` ignores a
  `stable_hash` field if a legacy snapshot still carries one, so old
  branches verify unchanged. Drift-detection power is identical: any body
  divergence between the committed file and the compiled manifests is still
  a `DRIFT` violation.
- The removal changed no hash value: the recompiled snapshot's hash before
  and after dropping the line is the same digest.

Mechanisms evaluated and REJECTED (evidence):

- **`.gitattributes merge=union`** — verified locally: a three-way merge of
  two branches with differing `stable_hash` lines keeps BOTH hunks →
  duplicate keys + a missing comma → `json.decoder.JSONDecodeError`. Union
  is line-append semantics; JSON is not. (It stays correct for
  `telemetry/model-usage.jsonl`, which IS line-append —
  `telemetry-merge-conflicts.md`.)
- **A custom git merge driver** — needs `git config` on every clone (no
  repo bootstrap installs one) and GitHub's server-side merge machinery
  never runs repo-defined drivers (verified live on this repo 2026-07-12,
  scratch PR #315 — see `telemetry-merge-conflicts.md`).

## Residual conflicts — the one recipe

Two concurrent PRs can still REALLY conflict in the snapshot body when
their hunks collide (e.g. both add refs to the same subsystem block). The
snapshot is generated — **never hand-resolve conflict markers in it**. From
the PR's branch checkout:

```
git fetch origin main
git merge origin/main          # resolve any SOURCE conflicts (sb/manifest, sb/domain) on their merits
git checkout --theirs manifest.snapshot.json   # any snapshot resolution is fine — it gets overwritten
python3 tools/manifest_compile.py --write      # regenerate from the merged sources
git add manifest.snapshot.json
git commit
```

The `--write` recompile from the merged sources is the ONLY correct
resolution; CI's P9 verify (`python3 tools/manifest_compile.py` in the `ci`
checkers job and the `manifest-validate` named gate) reds any snapshot that
does not match a fresh recompile.
