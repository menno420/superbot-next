## Re-arm recipe (for a fresh coordinator)

1. **Chain**: `create_trigger` one-shot (`run_once_at` ~16 min out,
   `persistent_session_id` = the coordinator session's own id) with the continuation
   prompt; re-arm a fresh one-shot each turn — a fired one-shot self-disables.
2. **Failsafe**: `create_trigger` recurring cron `0 */2 * * *`
   (`persistent_session_id` same session) with the failsafe prompt (verbatim text in
   the ORDER 008 record, control/status.md).
3. Rule text lives durably in docs/collaboration-model.md § Continuous mode (Q-0265);
   the exact original calls + outcomes in control/status.md's ORDER 008 record.

## Lessons

- One-shot chains survive turn-length variance better than a single recurring cron,
  but they die silently with the environment — the 2-hourly failsafe is what made the
  16:31Z teardown recoverable (for this lane) instead of fatal.
- Fleet-wide trigger state is invisible from inside one repo: the teardown's blast
  radius on sibling lanes only surfaced because the coordinator checked
  `list_triggers` after its own failsafe died. A periodic fleet-level trigger audit
  (manager seat) would catch this class without luck.
